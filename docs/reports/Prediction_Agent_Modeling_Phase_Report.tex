\chapter*{Executive Summary}
\addcontentsline{toc}{chapter}{Executive Summary}

This report presents a comprehensive technical analysis of the \textbf{QoS Prediction Agent (QoS Buddy)}, a production-grade machine learning system for predicting telecom Quality of Service (QoS) degradation events. The system is designed to assist Network Operations Center (NOC) operators in proactively detecting and responding to network incidents affecting voice, data, and capacity metrics.

The system represents a \textbf{multi-model ensemble approach} combining:
\begin{itemize}
    \item \textbf{XGBoost classifiers} (6 per-target models for snapshot risk detection)
    \item \textbf{LSTM recurrent network} (multi-target temporal pattern recognition)
    \item \textbf{Prophet forecaster} (capacity exhaustion ETA calculation)
    \item \textbf{SHAP explainability} (feature-level interpretability)
    \item \textbf{Retrieval-Augmented Generation (RAG)} with ChromaDB (incident context)
    \item \textbf{LLM integration} via Ollama (human-readable alert generation)
\end{itemize}

The entire pipeline follows \textbf{TDSP (Team Data Science Process)} methodology with rigorous emphasis on reproducibility, interpretability, and operational readiness.

\chapter{Project Architecture \& Design Philosophy}

\section{System Objective}

The QoS Prediction Agent addresses a critical operational challenge in telecom networks: \textbf{predicting when Quality of Service will degrade within a 60-minute horizon}. Rather than reacting to user complaints (which arrive after damage), the system enables proactive mitigation by:

\begin{enumerate}
    \item \textbf{Risk quantification} across 6 clinically distinct failure modes
    \item \textbf{Root cause identification} via SHAP explanations
    \item \textbf{Historical context retrieval} from similar past incidents
    \item \textbf{Operator guidance} in natural language via LLM summaries
    \item \textbf{Capacity forecasting} to estimate when network exhaustion will occur
\end{enumerate}

\section{Design Principles}

The system is built on three core principles:

\subsection{Principle 1: No Temporal Leakage}

All models are trained exclusively on \textbf{past data} to predict \textbf{future events}. Strictly enforced via:
\begin{itemize}
    \item TimeSeriesSplit (no shuffle; respects temporal order)
    \item Feature windows that only include historical context
    \item Explicit leakage detection in label and feature engineering
    \item Per-node isolation to prevent cross-cell contamination
\end{itemize}

\subsection{Principle 2: Explainability Over Accuracy}

While predictive accuracy matters, \textbf{operational utility} requires interpretability:
\begin{itemize}
    \item SHAP values explain every prediction at feature level
    \item Severity bands (normal/watch/warning/high/critical) provide clear decision boundaries
    \item Per-target margins make severity calculations transparent
    \item Retrieved incidents add historical context
    \item LLM alerts explain reasoning in business terminology
\end{itemize}

\subsection{Principle 3: Robustness Over Elegance}

Code prioritizes defensive programming and debugging visibility:
\begin{itemize}
    \item Explicit error logging at every critical step
    \item Validation of data shapes, ranges, and types
    \item Graceful fallbacks when models fail
    \item Class imbalance handling with explicit diagnostics
    \item Redundant checks to prevent silent failures
\end{itemize}
\section{System Architecture}

The system follows a modular pipeline architecture:

\begin{itemize}
    \item \textbf{Raw QoS Data} $\rightarrow$ Schema Cleaning
    \item Schema Cleaning $\rightarrow$ Preprocessing (encoding + imputation)
    \item Preprocessing $\rightarrow$ Feature Engineering (45+ columns)
    \item Feature Engineering $\rightarrow$ Label Engineering (6 targets, 120-step horizon)
    \item Label Engineering $\rightarrow$ Model Training (XGBoost, LSTM, Prophet, TTE)
    \item Model Training $\rightarrow$ Ensemble Fusion (0.55 XGB + 0.45 LSTM)
    \item Ensemble Fusion $\rightarrow$ Evaluation (ROC-AUC, Average Precision)
    \item Evaluation $\rightarrow$ Inference (SHAP, RAG, LLM alerts)
\end{itemize}

\chapter{Feature Engineering}

\section{Overview \& Philosophy}

Module: \texttt{data\_pipeline/features.py}

The feature engineering stage adds \textbf{45+ derived columns} that capture temporal patterns, interactions, and domain-specific metrics.

\subsection{Core Principle: No Leakage}

All features are \textbf{strictly backward-looking} within each \texttt{node\_id}:
\begin{itemize}
    \item Lags use \texttt{.shift(k)} to pull historical values
    \item Rolling windows use \texttt{rolling(window, min\_periods=...)} on past rows only
    \item Linear regression slopes computed only from past points
    \item Interactions combine only historical/derived features
\end{itemize}

\section{Engineered Features (45+)}

\subsection{Category 1: Composite Risk Indices}

\begin{enumerate}
    \item \textbf{congestion\_index} = queue\_length / (active\_connections + 1)
        \begin{itemize}
            \item Queuing theory ratio
            \item Used by Prophet for capacity forecasting
        \end{itemize}

\item \textbf{signal\_instability\_index} = std(rsrp\_dbm, rsrq\_db, sinr\_db)
        \begin{itemize}
            \item Multi-dimensional radio signal variance
            \item High values indicate fluctuating signal
        \end{itemize}

\item \textbf{coverage\_risk\_score} = (1 - norm(rsrp)) + (1 - norm(sinr)) + (1 - norm(cqi)) / 3
        \begin{itemize}
            \item Normalized combination of three radio metrics
            \item Averaged degradation across dimensions
        \end{itemize}

\item \textbf{voice\_qoe\_score} = (norm(jitter) + norm(packet\_loss) + (1 - norm(mos))) / 3
        \begin{itemize}
            \item Voice Quality of Experience composite
            \item Jitter/packet loss increase risk; MOS decreases it
        \end{itemize}
\end{enumerate}

\subsection{Category 2: Temporal Dynamics (Lags \& Trends)}

\begin{enumerate}
    \item \textbf{anomaly\_score lags} (lag1, lag3, lag5)
    \item \textbf{anomaly\_score rolling stats} (rmean5, rmean10, rmax10, rstd5)
        \begin{itemize}
            \item \textbf{Cross-correlation}: \texttt{anomaly\_score\_rmean5} $\rightarrow$ \texttt{call\_drop\_risk}: +0.615 (strongest signal)
        \end{itemize}
    \item \textbf{Metric lags}: \texttt{latency\_ms}, \texttt{jitter\_ms}, \texttt{throughput\_mbps}, \texttt{packet\_loss\_pct} (lag1, lag3, lag5)
\end{enumerate}

\subsection{Category 3: Anomaly Type Indicators}

Binary flags parsed from \texttt{anomaly\_type}:
\begin{itemize}
    \item \texttt{is\_weak\_signal}, \texttt{is\_packet\_loss}, \texttt{is\_high\_latency}
    \item \texttt{is\_jitter\_deg}, \texttt{is\_congestion\_evt}, \texttt{is\_high\_jitter}
    \item Rolling rates (e.g., \texttt{is\_high\_latency\_rate10}): fraction of recent timesteps with anomaly
\end{itemize}

\subsection{Category 4: Interaction Features}

\begin{itemize}
    \item \texttt{jitter\_x\_latency\_stress} = jitter\_rolling\_mean $\times$ latency\_rolling\_mean / 100
    \item \texttt{rsrp\_x\_sinr} = rsrp\_dbm $\times$ sinr\_db
    \item \texttt{throughput\_sinr\_product} = throughput\_mbps $\times$ sinr\_db
    \item \texttt{memory\_connections\_ratio} = memory\_pct / (active\_connections + 1)
    \item \texttt{volatility\_combined} = throughput\_volatility + jitter\_rolling\_std
\end{itemize}

\subsection{Category 5: Slope Features (Temporal Trends)}

\begin{enumerate}
    \item \textbf{rsrp\_slope\_5min} = linear regression slope of rsrp\_dbm over 10-step window
    \item \textbf{mos\_trend\_slope} = linear regression slope of mos\_estimate over 10-step window
\end{enumerate}

\chapter{Label Engineering \& Temporal Strategy}

\section{Future-Window Labels}

Module: \texttt{data\_pipeline/label\_engineer.py}

Standard supervised learning is insufficient for operational QoS. The system requires:

\begin{quote}
\textit{``Given today's features, will QoS degrade within 60 minutes?''}
\end{quote}

\subsection{Label Architecture}

\begin{verbatim}
Current time: t
Future window: [t+1, t+120] (120 steps × 30 sec = 60 minutes)

Feature vector at time t:
  - latency_ms, throughput_mbps, jitter_ms (current + lags)
  - anomaly_score, rolling stats
  → input to model

Label at time t:
  - Did latency exceed 95ms in [t+1, t+120]? → latency_breach_risk
  - Did throughput drop below 3 Mbps? → throughput_risk
  - Did anomaly_score exceed 0.85? → call_drop_risk
  - ... (6 targets total)
\end{verbatim}

\section{Six Target Definitions}

All targets use a \textbf{120-step horizon} (60 minutes) but employ different aggregation strategies:

\subsection{Target 1: call\_drop\_risk (Rolling MAX)}

\begin{equation*}
\text{call\_drop\_risk}[t] = \begin{cases}
1 & \text{if } \max(\text{anomaly\_score}[t+1:t+120]) > 0.85 \\
0 & \text{otherwise}
\end{cases}
\end{equation*}

\textbf{Rationale}: Single spike is sufficient signal. Most aggressive target (easiest to trigger).

\subsection{Target 2: latency\_breach\_risk (Rolling MEAN)}

\begin{equation*}
\text{latency\_breach\_risk}[t] = \begin{cases}
1 & \text{if } \text{mean}(\text{latency\_ms}[t+1:t+120]) > 95.0 \\
0 & \text{otherwise}
\end{cases}
\end{equation*}

\textbf{Rationale}: Latency breaches harm when sustained. MEAN captures degradation, not transient spikes.

\subsection{Target 3: throughput\_risk (Rolling MIN)}

\begin{equation*}
\text{throughput\_risk}[t] = \begin{cases}
1 & \text{if } \min(\text{throughput\_mbps}[t+1:t+120]) < 3.0 \\
0 & \text{otherwise}
\end{cases}
\end{equation*}

\textbf{Rationale}: Captures worst sustained degradation.

\subsection{Target 4: jitter\_risk (Rolling MEAN)}

Similar to latency---VoIP quality requires sustained low jitter.

\subsection{Target 5: congestion\_risk (Rolling MAX)}

\begin{equation*}
\text{congestion\_index} = \frac{\text{queue\_length}}{\text{active\_connections} + 1}
\end{equation*}

\begin{equation*}
\text{congestion\_risk}[t] = \begin{cases}
1 & \text{if } \max(\text{congestion\_index}[t+1:t+120]) > 0.85 \\
0 & \text{otherwise}
\end{cases}
\end{equation*}

\subsection{Target 6: mos\_risk (Rolling MIN)}

\begin{equation*}
\text{mos\_risk}[t] = \begin{cases}
1 & \text{if } \min(\text{mos\_estimate}[t+1:t+120]) < 3.0 \\
0 & \text{otherwise}
\end{cases}
\end{equation*}

\textbf{Rationale}: Mean Opinion Score must stay above 3.0 for acceptable service.

\subsection{Why XGBoost?}

\begin{enumerate}
    \item \textbf{Feature importance}: Tree-based splits reveal which features matter
    \item \textbf{Non-linearity}: Captures complex relationships
    \item \textbf{Mixed features}: Works with categorical-encoded and continuous inputs
    \item \textbf{Production-proven}: Industry-standard, robust, well-understood
    \item \textbf{Speed}: Fast training and inference
\end{enumerate}

\subsection{Class Imbalance Handling}

\begin{lstlisting}
def compute_balanced_scale_pos_weight(y_true):
    """
    scale_pos_weight = n_negatives / n_positives

E.g., 2% positive samples:
    scale_pos_weight = (0.98 / 0.02) ≈ 49

Clamp to [0.5, 10.0] to prevent instability.
    """
    n_pos = (y_true == 1).sum()
    n_neg = (y_true == 0).sum()

if n_pos == 0:
        logger.error("Zero positives; cannot compute scale_pos_weight")
        return 1.0

raw_spw = n_neg / n_pos
    clamped = np.clip(raw_spw, 0.5, 10.0)

logger.info(f"scale_pos_weight={clamped:.2f}")
    return clamped
\end{lstlisting}

\textbf{Why clamping?} Extreme values destabilize training. Clamping to [0.5, 10.0] provides safe range.

\subsection{Calibration: Isotonic Regression}

Raw XGBoost scores are not well-calibrated probabilities. Solution: wrap with \texttt{CalibratedClassifierCV}:

\begin{lstlisting}
# Fit base model
xgb = XGBClassifier(..., scale_pos_weight=spw)
xgb.fit(X_train, y_train)

# Wrap with isotonic calibration
calibrated = CalibratedClassifierCV(xgb, cv='prefit')
calibrated.fit(X_train, y_train)

# Apply to test
y_proba = calibrated.predict_proba(X_test)[:, 1]
\end{lstlisting}

\textbf{Benefit}: Confidence scores now match empirical frequencies.

\section{LSTM: Multi-Label Temporal Model}

Module: \texttt{models/lstm\_trainer.py}

\subsection{Architecture}

Single LSTM network predicts all 6 targets simultaneously:

\begin{verbatim}
Input: (batch_size, window_length=20, n_features=45)
  ↓
LSTM Layer 1: 128 hidden units, dropout=0.3
  ↓
LSTM Layer 2: 64 hidden units, dropout=0.3
  ↓
Dense Layer: 64 hidden units, ReLU, dropout=0.2
  ↓
Output Layer: 6 neurons (sigmoid for multi-label)
  ↓
Output: (batch_size, 6) probabilities in [0, 1]
\end{verbatim}

\subsection{Why LSTM?}

\begin{enumerate}
    \item \textbf{Temporal modeling}: Captures sequential patterns
    \item \textbf{Shared representation}: Single model for all 6 targets
    \item \textbf{Multi-label}: Learns cross-target dependencies
    \item \textbf{Complementary}: XGBoost = snapshot; LSTM = trend detection
\end{enumerate}

\subsection{Window Construction}

\begin{lstlisting}
def _build_windows(df, feature_cols, window=20):
    """
    For each timestep t (from window to end):
        Window = features[t-window : t]    (20 past rows)
        Label  = labels[t]                 (future label)

No cross-node mixing, no leakage.
    """
    X_list, y_list = [], []

for node_id, g in df.groupby("node_id", sort=False):
        g = g.reset_index(drop=True)

if len(g) <= window:
            continue  # Skip small nodes

feats = g[feature_cols].to_numpy(dtype=np.float32)
        labels = g[TARGET_NAMES].to_numpy(dtype=np.float32)

for t in range(window, len(g)):
            X_list.append(feats[t-window:t, :])
            y_list.append(labels[t, :])

return np.stack(X_list), np.vstack(y_list)
\end{lstlisting}

\subsection{Scaler Fit on Training Windows Only}

\textbf{Critical}: Fit MinMaxScaler only on training windows to prevent leakage:

\begin{lstlisting}
# Extract training windows
X_train_flat = X_train.reshape(-1, n_features)

# Fit scaler ONLY on training
scaler = MinMaxScaler()
scaler.fit(X_train_flat)

# Apply to both train and test
X_train_scaled = scaler.transform(X_train.reshape(-1, n_features))
X_test_scaled = scaler.transform(X_test.reshape(-1, n_features))
\end{lstlisting}

\textbf{Why?} If scaler is fit on train+test, test statistics leak into scaling.

\section{Prophet: Capacity Exhaustion ETA}

Module: \texttt{models/prophet\_forecaster.py}

\subsection{Purpose}

Prophet forecasts when \texttt{congestion\_index} will exceed 0.85 (capacity exhaustion). Separate from binary classification.

\subsection{Architecture}

\begin{itemize}
    \item \textbf{Per-node models}: One Prophet model per \texttt{node\_id}
    \item \textbf{Target metric}: \texttt{congestion\_index}
    \item \textbf{Forecast horizon}: 120 periods (60 minutes)
    \item \textbf{Components}: Trend, seasonality (optional)
\end{itemize}

\subsection{Why Separate from Ensemble?}

\begin{enumerate}
    \item \textbf{Different semantics}: Prophet = ``when will congestion happen?''; XGB/LSTM = ``is congestion happening?''
    \item \textbf{Time-series specific}: Prophet uses time-indexed data vs. feature vectors
    \item \textbf{Operational utility}: NOC operators want both answers independently
    \item \textbf{Failure independence}: If Prophet fails, core prediction works
\end{enumerate}

\section{ETA Regressors: Time-to-Event (TTE) Models}

Module: \texttt{models/eta\_trainer.py}

\subsection{Purpose}

ETA regressors predict the \textbf{time-to-event (TTE)} in minutes for 5 non-congestion targets. While the ensemble predicts binary risk (yes/no), the ETA models answer: \textit{``How many minutes until this specific degradation occurs?''}

\subsection{Design Rationale}

\begin{enumerate}
    \item \textbf{Event-conditional training}: Models train ONLY on rows where the target event occurred
    \item \textbf{Log-normal regression}: Uses \texttt{log1p} transformation to handle skewed TTE distributions
    \item \textbf{Per-target regressors}: Separate XGBRegressor for each of 5 targets
    \item \textbf{Operational value}: Provides concrete time estimates for mitigation planning
\end{enumerate}

\subsection{Target Definitions}

ETA models predict TTE for:

\begin{enumerate}
    \item \textbf{call\_drop\_risk\_tte}: Minutes until anomaly\_score exceeds 0.85
    \item \textbf{latency\_breach\_risk\_tte}: Minutes until sustained latency exceeds 95 ms
    \item \textbf{throughput\_risk\_tte}: Minutes until throughput drops below 3 Mbps
    \item \textbf{jitter\_risk\_tte}: Minutes until sustained jitter exceeds 20 ms
    \item \textbf{mos\_risk\_tte}: Minutes until Mean Opinion Score drops below 3.0
\end{enumerate}

\textbf{Note}: Congestion (6th target) uses Prophet forecaster instead.

\subsection{Training Architecture}

\begin{lstlisting}
def train_eta_models(df: pd.DataFrame) -> dict[str, EtaModelBundle]:
    """
    For each target in ETA_TARGETS:
    1. Filter rows where event occurred (event_column == 1)
    2. Extract feature_cols and TTE values
    3. Apply log1p transformation: y = log1p(tte)
    4. Train XGBRegressor on log-scale targets
    5. Store model + metadata (event_rate, median_tte, etc.)

    for target in ETA_TARGETS:

        event_col = EVENT_COLUMN_MAP[target]  # e.g., 'latency_breach_event'

# CRITICAL: Train ONLY on events
        train_df = df[df[event_col] == 1].copy()
        train_df = train_df.dropna(subset=[tte_col])

if train_df.empty:
            logger.error(f"No events for {target}; cannot train ETA model")
            continue

X = train_df[feature_cols].astype(np.float32)

reg = XGBRegressor(
            n_estimators=500,

            colsample_bytree=0.8,
            min_child_weight=4,
            reg_alpha=0.1,
            reg_lambda=1.5,
        )
        reg.fit(X, y)
        models[target] = EtaModelBundle(...)
\end{lstlisting}

\subsection{Key Implementation Details}

\textbf{Log-Normal Transform}:
\begin{itemize}
    \item Input TTE values (minutes) are naturally skewed: many short events, few long ones
    \item Solution: \texttt{y = log1p(tte)} compresses distribution
    \item Prediction: \texttt{eta = expm1(model.predict(X))}
    \item Prevents overflow: clamps unrealistic ETAs (> 1000 min)
\end{itemize}

\textbf{Event-Only Training}:
\begin{itemize}
    \item Only rows where \texttt{event == 1} contribute training signal
    \item Prevents false regression on non-events
    \item Requires minimum 3 events; warning if < 10 events
    \item Logs event rate per target (e.g., 2.3\% for call\_drop\_risk)
\end{itemize}

\textbf{Leakage Prevention}:
\begin{itemize}
    \item Validates feature list: blocks \texttt{tte\_*}, \texttt{*\_event}, \texttt{*\_risk} columns
    \item Raises \texttt{ValueError} on ANY leaky column detected
    \item Uses identical feature set as classification models
\end{itemize}

\subsection{Prediction Output}

\begin{lstlisting}
class EtaModelBundle:
    target: str                           # e.g., 'latency_breach_risk'
    model: XGBRegressor | None            # Trained regressor
    method: str                           # 'xgb_regressor_log1p_events_only'
    event_rate: float                     # % of rows with event (e.g., 0.057)
    median_tte_min: float                 # Median TTE for occurred events
    horizon_min: float                    # Minimum horizon (30 sec / 60 = 0.5 min)

def predict_minutes(self, row: pd.Series) -> float:
        """Return ETA in minutes, or inf if unavailable."""
        if self.model is None:
            return float("inf")

x = row[self.feature_cols].to_numpy().reshape(1, -1)
        log_scale = self.model.predict(x)[0]
        eta = np.expm1(log_scale)  # Inverse of log1p

# Sanity checks
        if not np.isfinite(eta) or eta < 0 or eta > 1000:
            return float("inf")

return eta
\end{lstlisting}

\subsection{When ETA is Unavailable}
The model returns \texttt{float("inf")} (no prediction) when:

\begin{itemize}
    \item Model is None (training failed for that target)
    \item Missing feature columns in input row
    \item Prediction is non-finite (NaN, inf)
    \item Expm1 overflow: log\_scale too large
    \item Negative ETA (model drift)
    \item ETA unreasonably large (> 1000 minutes $\approx$ 16 hours)
\end{itemize}

\subsection{ROC-AUC (Receiver Operating Characteristic - Area Under Curve)}

\textbf{Definition}: Probability that model ranks a random positive example higher than a random negative example.

\textbf{Range}: [0, 1] where 0.5 = random, 1.0 = perfect.

\textbf{Interpretation}: 
\begin{itemize}
    \item ROC-AUC = 0.75: Model 75\% likely to rank true incident higher than true non-incident
    \item Robust to class imbalance (unlike accuracy)
    \item Enables threshold tuning
\end{itemize}

\subsection{Average Precision (AP)}

\textbf{Definition}: Probability that predictions at decision threshold are correct, averaged over all thresholds.

\textbf{Range}: [0, 1] where 0 = never positive, 1 = perfect.

\textbf{Interpretation}:
\begin{itemize}
    \item AP = 0.60: Of all predictions labeled ``HIGH RISK'', 60\% are actually incidents
    \item Better aligned with operational goal: avoid false alarms
    \item More pessimistic than ROC-AUC for imbalanced data
\end{itemize}

\section{SHAP Feature Importance}

Module: \texttt{explainability/shap\_explainer.py}

SHAP (SHapley Additive exPlanations) computes each feature's contribution to a prediction:

\begin{lstlisting}
def explain_tabular_row(models, feature_names, x_row):
    """
    For each target:
    1. Get XGBoost model for that target
    2. Create TreeExplainer
    3. Compute SHAP values for input row
    4. Return top-5 features by absolute SHAP

Output: List of {target, feature, value, direction}
    """
    output = []
    for target_name in TARGET_NAMES:
        model = models[target_name]
        explainer = shap.TreeExplainer(model)
        shap_values = explainer.shap_values(x_row)

# Get top-5 by absolute SHAP
        top_indices = np.argsort(-np.abs(shap_values))[:5]

for idx in top_indices:
            direction = "increases_risk" if shap_values[idx] >= 0 \
                       else "decreases_risk"
            output.append({
                "target": target_name,
                "feature": feature_names[idx],
                "value": shap_values[idx],
                "direction": direction,
            })

return output
\end{lstlisting}

\subsection{Why SHAP?}

\begin{enumerate}

    \item \textbf{Model-agnostic}: Applicable to any model
\end{enumerate}

\section{Severity Mapping}

Probabilities mapped to severity bands using \textbf{margins} (probability - threshold):

\begin{lstlisting}
def _severity_from_margin(max_margin):
    """
    Threshold = 0.5 (standard decision boundary)
    Margin = probability - 0.5

Severity bands:
    - normal:   margin < -0.15
    - watch:    -0.15 ≤ m < -0.05
    - warning:  -0.05 ≤ m < 0.05
    - high:     0.05 ≤ m < 0.15
    - critical: m ≥ 0.15
    """
    if max_margin < -0.15:
        return "normal"
    elif max_margin < -0.05:
        return "watch"
    elif max_margin < 0.05:
        return "warning"
    elif max_margin < 0.15:
        return "high"
    else:
        return "critical"
\end{lstlisting}

\subsection{Examples}

\begin{itemize}
    \item Probability = 0.65 $\Rightarrow$ margin = 0.15 $\Rightarrow$ \textit{critical}
    \item Probability = 0.52 $\Rightarrow$ margin = 0.02 $\Rightarrow$ \textit{warning}
    \item Probability = 0.35 $\Rightarrow$ margin = -0.15 $\Rightarrow$ \textit{normal}
\end{itemize}

\chapter{Technical Decisions \& Trade-offs}

\section{TimeSeriesSplit vs. Random Shuffle}

\textbf{Decision}: Use TimeSeriesSplit(n\_splits=5)

\textbf{Why?}
\begin{itemize}
    \item Time-series temporal dependencies
    \item Realistic deployment (train past, test future)
    \item Prevents leakage
\end{itemize}

\textbf{Trade-off}: Smaller test folds. Mitigated by using last fold (most data-rich).

\section{Future-Window Labels (120-Step Horizon)}

\textbf{Decision}: Predict degradation in [t+1, t+120]

\textbf{Why?}
\begin{itemize}
    \item Operational relevance (60-min lead time)
    \item Preventive action window
    \item Business SLA requirement
\end{itemize}

\textbf{Trade-off}: Highly imbalanced labels (1--6\% positive). Handled via \texttt{scale\_pos\_weight}.

\section{Per-Target Feature Exclusion}

\textbf{Decision}: Remove features with sign-flipping correlation

\textbf{Why?}
\begin{itemize}
    \item Data distribution shift (baseline vs. anomaly periods)
    \item Model robustness improvement (3--5\% AUC gain)
    \item Prevents learning wrong feature direction
\end{itemize}

\textbf{Trade-off}: Domain-specific hack. Better solution: domain adaptation (future work).

\section{Ensemble Weights (0.55 XGB, 0.45 LSTM)}

\textbf{Decision}: Slightly favor XGBoost

\textbf{Why?}
\begin{itemize}
    \item XGBoost faster, production-proven
    \item LSTM captures complementary temporal patterns
    \item Close weights (not 0.80:0.20) ensure both contribute
\end{itemize}

\textbf{Trade-off}: Hyperparameters; could be tuned via cross-validation.

\section{Prophet Never Blended}

\textbf{Decision}: Use Prophet ETA separately

\textbf{Why?}
\begin{itemize}
    \item Different objective (``when'' vs. ``if'')
    \item Operational utility (independent answers)
    \item Failure isolation
\end{itemize}

\textbf{Trade-off}: Miss potential ensemble gains. Mitigated by interpretable ETA.

\section{Ollama (Local LLM) vs. Cloud}

\textbf{Decision}: Use Ollama

\textbf{Why?}
\begin{itemize}
    \item Cost-free (no API charges)
    \item Privacy (data stays in-house)
    \item Speed (sub-second latency)
    \item Control (any open-source model)
\end{itemize}

\textbf{Trade-off}: Lower quality than GPT-4/Claude. Mitigated by prompt engineering.

% ==================== CHAPTER 12: LIMITATIONS ====================
\chapter{Limitations \& Future Work}

\section{Limitations}

\subsection{1. Class Imbalance}
Only 1--6\% positive labels make alerts naturally low-precision. Future: SMOTE, cost-sensitive learning.

\subsection{2. Single Network Domain}
Models trained on one telecom network. May not generalize to different infrastructure.

\subsection{3. Data Distribution Shift}
Feature correlations differ between training (baseline) and test (anomaly) periods. Requires per-target feature exclusion (inelegant).

\subsection{4. LSTM Limited Window}
20-step windows miss longer-term patterns (hours). Future: hierarchical models, attention mechanisms.

\subsection{5. Ollama Quality}
Local LLMs (1B--7B params) lower quality than GPT-4. Prompts must be very precise.

\subsection{6. No Online Learning}
Models retrained in batch (offline). Cannot adapt to new failure modes in real-time.

\section{Future Enhancements}

\subsection{1. Causal Inference}
Identify root causes vs. correlated symptoms using causal graphs.

\subsection{2. Federated Learning}
Train on multiple networks without centralizing sensitive data.

\subsection{3. Multi-Task Learning}
Jointly predict risk, ETA, and anomaly type; share representations.

\subsection{4. Hierarchical Forecasting}
Predict failures at node, cell, network levels; reconcile contradictions.

\subsection{5. Active Learning}
Identify high-uncertainty predictions; prioritize for manual labeling.

\subsection{6. Cost-Sensitive Classification}
Assign different misclassification costs (false alert vs. missed alert).

\subsection{7. Temporal Point Processes}
Model event timing explicitly (when will failures cluster?).

\chapter{Glossary}

\begin{description}
    \item[AUC] Area Under Curve (ROC-AUC); measures ranking quality
    \item[AP] Average Precision; emphasizes correct positive predictions
    \item[Calibration] Mapping model outputs to true probabilities
    \item[ETA] Estimated Time-to-event (minutes)
    \item[LSTM] Long Short-Term Memory recurrent network
    \item[MoS] Mean Opinion Score (voice quality, 1--5)
    \item[NOC] Network Operations Center
    \item[OOV] Out-of-vocabulary (unseen category)
    \item[Prophet] Facebook's time-series forecasting library
    \item[QoS] Quality of Service (network metrics)
    \item[RAG] Retrieval-Augmented Generation
    \item[SHAP] SHapley Additive exPlanations
    \item[TDSP] Team Data Science Process
    \item[TimeSeriesSplit] Temporal cross-validation preserving order
    \item[TTE] Time-to-event (minutes until threshold breach)
    \item[XGBoost] eXtreme Gradient Boosting
\end{description}
